How to fix — write a strong description
Each component description doubles as the trigger the model reads to decide whether to invoke it. Vague descriptions block routing more often than they block security — that's why this is enforced.
Pattern cheatsheet
  • Skills: Use when <trigger condition> — <what it does>. The description IS the trigger string Claude reads to decide whether to invoke the skill, so name the condition first.
  • Agents: <What the agent does>. Use for <invocation context>. The routing layer uses this string; describe the dispatch criterion.
  • Plugins: one-sentence marketplace pitch — <verb> <noun> for <audience>. Treat it like the elevator pitch on the marketplace tile.
  • Commands: one-verb summary of the action — "Run the test suite and print failures grouped by module."
What "good enough" means
  • At least 30 characters (20 for commands)
  • At least 4 distinct words
  • Action-oriented; names the trigger condition
  • No TODO / TBD / template placeholders
  • Specific (names the domain, tech, or scenario)
