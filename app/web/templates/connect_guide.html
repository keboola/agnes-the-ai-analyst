{{ instance_brand }} exposes a native OAuth 2.1 remote MCP connector. Any MCP-compatible AI agent connects to it through a standard browser-based sign-in — there is no personal access token to copy, no client secret to manage. Each agent registers itself automatically, opens a browser for you to authenticate, and receives a short-lived token scoped to your account.

Everything an agent does is filtered by your permissions: the connector's token maps to your {{ instance_brand }} user, so every table, schema, and query obeys the same RBAC rules that apply everywhere else in the product.

{# Claude Desktop #}

Claude Desktop

Requires the Claude Desktop app (macOS or Windows), signed in to your Claude account.

  1. Open Settings → Connectors and click Add custom connector.
  2. Paste your connector URL (<your-host>/api/mcp/http) into the URL field and click Add.
  3. Claude registers itself automatically and opens a browser window.
  4. Sign in to {{ instance_brand }} with your usual identity provider (e.g. Google, or the email magic-link fallback).
  5. On the consent screen, review the requested client name and scope, then click Allow.
  6. The connector goes live — the {{ instance_brand }} tools are now available in your conversations.
{# Claude.ai #}

Claude.ai (web)

Requires a Claude.ai plan that supports custom connectors.

  1. Go to Settings → Connectors and choose Add custom connector.
  2. Paste your connector URL (<your-host>/api/mcp/http) and confirm.
  3. Complete the browser sign-in and approve the consent screen.
  4. Enable the connector in a chat to expose the tools to Claude.
{# Cursor #}

Cursor

Requires a recent Cursor build with MCP support.

1. Add the server

Create or edit .cursor/mcp.json (per-project) or ~/.cursor/mcp.json (global):

{
  "mcpServers": {
    "agnes": {
      "url": "https://<your-host>/api/mcp/http"
    }
  }
}

2. Sign in

  1. Open Settings → MCP; the agnes server appears in the list.
  2. Click the sign-in / refresh action next to it — a browser opens for the {{ instance_brand }} login and consent.
  3. Once authorized, the tools are available in Agent and Composer.
{# VS Code / Copilot #}

VS Code / GitHub Copilot

Requires VS Code 1.101+ with the GitHub Copilot Chat extension (native MCP with OAuth). Update both to the latest version if the sign-in does not appear.

Option A — add via the Command Palette (recommended)

  1. Open the Command Palette (⇧⌘P / Ctrl+Shift+P) and run MCP: Add Server.
  2. Choose HTTP (HTTP or Server-Sent Events).
  3. Paste your connector URL (<your-host>/api/mcp/http) and name the server agnes.
  4. Choose Global to use it in every workspace (or Workspace to scope it to the current project).
  5. The server starts and opens a browser — sign in to {{ instance_brand }} and click Allow.

Option B — commit it to the workspace

Add .vscode/mcp.json to share the server with everyone who opens the project:

{
  "servers": {
    "agnes": {
      "type": "http",
      "url": "https://<your-host>/api/mcp/http"
    }
  }
}

.vscode/mcp.json

Use it in Copilot

  1. Open Copilot Chat and switch the mode selector to Agent.
  2. Click the tools icon (🔧) and enable agnes.
  3. Verify the connection by asking the agent to run server_info — it should return your account email.
If sign-in fails or the server gets stuck, run MCP: Clear Saved Authorizations from the Command Palette, remove and re-add the server, then restart VS Code. A stale cached registration is the most common cause.
{# ChatGPT #}

ChatGPT connectors

Requires a ChatGPT plan that supports custom / remote MCP connectors.

  1. Open the connector settings and choose to add a remote MCP server.
  2. Paste your connector URL (<your-host>/api/mcp/http).

{% endcall %} {# ── How it works (always visible) ─────────────────────────────── #} {% call ds.panel(title='How it works', tag='section', attrs='aria-label="How it works"') %}

The browser handshake every agent performs:

  1. The agent calls the connector URL and receives 401 with a pointer to the discovery metadata.
  2. It reads the protected-resource and authorization-server metadata (RFC 9728 / RFC 8414).
  3. It registers itself automatically (RFC 7591 dynamic client registration).
  4. It opens the browser at the authorize endpoint using PKCE; you sign in to {{ instance_brand }} and approve the consent screen.
  5. It exchanges the authorization code for an access token, then sends Authorization: Bearer <token> on every call.

The access token is a standard {{ instance_brand }} session token (8-hour lifetime), renewable via the issued refresh token (30 days) without re-authorizing. Tokens can be revoked at any time and take effect on the next call.
