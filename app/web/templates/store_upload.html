Every component description (plugin, agents, skills, commands) is the trigger string Claude reads to decide whether to invoke it. Vague or missing descriptions get rejected even when the code is fine.

The bar:

  • Each description ≥ {{ guardrail.min_description_chars|default(60) }} characters (commands ≥ {{ guardrail.min_command_description_chars|default(25) }}). Aim for one complete sentence — too short and the assistant can't tell when to use it.
  • At least {{ guardrail.min_distinct_words|default(5) }} distinct words
  • No TODO / TBD / template placeholders
  • Action-oriented, names the trigger condition

Patterns that work:

  • Skills: Use when <trigger> — <what it does>
  • Agents: <What the agent does>. Use for <invocation context>
  • Plugins: one-sentence marketplace pitch
  • Commands: one-verb summary of the action

The reviewer also runs a substantive pass on descriptions — generic ones ("a useful skill for working with data") get flagged even when they clear the mechanical bar.
