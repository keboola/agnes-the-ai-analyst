## CRITICAL — plan blockers
1. Phase 0 is a self-blocking lint gate
   - finding: The plan says Phase 0 ships a lint rule before the cleanup PRs, but the same section says existing violations fail until lifted. That makes Phase 1 unable to merge behind Phase 0 unless CI is intentionally red or the lint rule has a baseline/new-code-only mode.
   - evidence: `docs/planning/orm-state-migration.md:219` says the rule fails on `conn.execute(<string-literal>)` or `sa.text(<string-literal>)`; `docs/planning/orm-state-migration.md:232` says "Existing 65 spots fail until lifted into repos"; the sequencing table puts Phase 1/2/3/4/5 dependencies on `0` at `docs/planning/orm-state-migration.md:448-454`; the verdict still says "Lint catches future drift | YES — Phase 0 ships standalone" at `docs/planning/orm-state-migration.md:474`.
   - impact: The first "small PR" either breaks CI immediately or cannot enforce what the plan claims. Quick-win PRs cannot merge under the stated dependency graph.
   - what the plan should say instead: Phase 0 must be a baseline rule: fail only new/changed callsites, or check against an explicit current-violations snapshot that shrinks per PR. Full enforcement can only become blocking after Phases 1-5 remove the known state-SQL hits.

2. `duckdb-engine` + `RETURNING` is not a validated foundation
   - finding: The plan requires SQLAlchemy-on-DuckDB to handle CAS-style `RETURNING`, but DuckDB's current UPDATE docs do not document `UPDATE ... RETURNING`, and `duckdb_engine` warns that it inherits PostgreSQL behavior that DuckDB may not support. This is a hard dependency, not a routine spike checkbox. [UNVERIFIABLE without python] Actual SQLAlchemy `update(...).returning(...)` execution was not tested because python was forbidden.
   - evidence: The plan's Phase 7 explicitly says "Validate: ... `RETURNING` clauses work for CAS patterns" at `docs/planning/orm-state-migration.md:337-342`. DuckDB UPDATE docs at `https://duckdb.org/docs/current/sql/statements/update` list UPDATE examples/syntax/upsert at lines 478-573 but have no `RETURNING` section; by contrast INSERT documents "The `RETURNING` clause may be used..." at `https://duckdb.org/docs/current/sql/statements/insert` lines 715-718 and DELETE documents "The `RETURNING` clause allows returning the deleted values" at `https://duckdb.org/docs/current/sql/statements/delete` lines 494-496. DuckDB issue `https://github.com/duckdb/duckdb/issues/9915` lines 212-214 says `update returning` "seems not to be documented/defined". `duckdb_engine` README warns: "SQLAlchemy may try to use PostgreSQL-only features" at `https://github.com/Mause/duckdb_engine` lines 349-354.
   - impact: CAS migrations in auth/token flows can silently become backend-specific, fail at runtime, or require hand-written SQL anyway. If the ORM path needs UPDATE RETURNING and DuckDB cannot support it through the dialect, the unified repo design collapses back into dialect branches.
   - what the plan should say instead: Make Phase 7 the first blocker before any ORM consolidation estimate is accepted. Add explicit pass/fail criteria for `insert returning`, `update returning`, `delete returning`, ORM flush defaults, rowcount behavior, and CAS alternatives that do not require UPDATE RETURNING.

3. Usage telemetry creates a split-brain state store
   - finding: The plan recommends keeping `usage_*` in DuckDB under a PG-state world, but the current processor reads lookup tables from the DuckDB cursor while writing events and summaries through `usage_repo()`, which dispatches to the active backend. That is exactly the mixed-backend condition the factory claims does not happen.
   - evidence: The plan recommends "Keep `usage_*` on DuckDB even under PG-state world" at `docs/planning/orm-state-migration.md:324-330`. `UsageProcessor.process_session` accepts a `duckdb.DuckDBPyConnection` at `services/session_processors/usage.py:33-39`, builds `MarketplaceItemLookup(conn)` at `services/session_processors/usage.py:53`, then writes via `repo = usage_repo()` at `services/session_processors/usage.py:103-105`. `MarketplaceItemLookup` reads `marketplace_plugins` and `store_entities` from that cursor at `services/session_processors/usage_lib.py:278-293`; rollups read the same lookup tables and write `usage_*` through the same cursor at `services/session_processors/usage_lib.py:650-660` and `services/session_processors/usage_lib.py:670-742`. The factory maps `usage` to both DuckDB and PG at `src/repositories/__init__.py:282-285` while claiming "We DO NOT mix backends within a request" at `src/repositories/__init__.py:23-26`.
   - impact: Under PG state, events can be written to PG while lookup attribution and rollups read/write DuckDB, or vice versa. Admin usage exports, rollups, and marketplace attribution can disagree about the source of truth.
   - what the plan should say instead: Decide one owner for `usage_*` before allowing PG state. Either make usage fully PG when `use_pg()` is true, or explicitly inject a DuckDB-only usage repository and a DuckDB lookup boundary, with tests proving no path writes usage rows to PG in that mode.

4. Retiring `src/db.py` is framed as DDL deletion, but it is operational recovery code
   - finding: The plan treats Phase 10 as deleting the DuckDB schema ladder after Alembic takes over, but `src/db.py` owns WAL replay recovery, pre-migration snapshots, idempotent backfills, and checkpoint safety. Alembic does not replace those operational behaviors.
   - evidence: The plan says "Delete `_v1_to_v(N)` chain from `src/db.py`" and "`src/db.py` shrinks from 5565 LOC to ~1500" at `docs/planning/orm-state-migration.md:406-416`; `wc -l src/db.py` returned `5565 src/db.py`. `src/db.py:1280-1390` implements WAL replay fallback and pre-migrate snapshot restore; `src/db.py:1397-1436` implements WAL salvage by discarding an unreplayable WAL; `src/db.py:5116-5127` checkpoints and copies `system.duckdb.pre-migrate`; `src/db.py:5453-5477` forces post-migration `CHECKPOINT` to avoid WAL replay failures.
   - impact: A naive ladder deletion can remove crash recovery and rollback protections for DuckDB state, especially during the exact migration windows this plan expands.
   - what the plan should say instead: Split Phase 10 into a design doc and implementation: inventory every non-DDL behavior in `src/db.py`, keep WAL/snapshot/checkpoint helpers until DuckDB state is genuinely no longer supported, and only delete specific migration functions after equivalent tests exist.

## MEDIUM — concerns to address
10. The lint rule misses the kind of SQL the repo already uses
   - finding: The exact false-positive search requested found no module-level uppercase `SQL`/`QUERY` constants, but the larger lint problem is false negatives: the proposed AST rule only catches literal arguments, while existing code builds SQL in variables and f-strings before executing it.
   - evidence: `rg -n "^[A-Z_]*SQL\\s*=\\|^[A-Z_]*QUERY\\s*=" src/ app/ -g "*.py" -l` returned no files. The plan's rule only fails on `conn.execute(<string-literal>)` or `sa.text(<string-literal>)` at `docs/planning/orm-state-migration.md:219`. `app/api/admin_usage.py:93-96` builds `sql` and `cnt_sql` variables, then executes `cnt_sql`; `app/api/admin_usage.py:125-128` executes a passed `sql` variable; `app/api/admin_usage.py:175-177` builds and executes `copy_sql`; `app/api/admin_usage.py:257-286` executes LLM-produced `validated_sql`.
   - impact: Phase 0 can pass while new raw SQL lands through local variables, f-strings, helper functions, or LLM-validated SQL paths. The invariant is not actually locked.
   - what the plan should say instead: Define the lint target as DBAPI/SQLAlchemy execution outside approved modules, not only literal arguments. Track tainted strings, f-strings, and variables, or explicitly baseline every dynamic SQL escape with a named allowlist reason.

11. DuckDB connection pooling and concurrency are underspecified
   - finding: The plan marks DuckDB single-writer + SQLAlchemy pool configuration as low likelihood and "already understood", but it does not specify the actual engine factory. The existing PG engine uses a multi-connection pool, and DuckDB's concurrency model has sharp write-conflict boundaries.
   - evidence: The risk table says "DuckDB single-writer + SA session pool | Low | Medium | SA pool config (`NullPool`) — already understood pattern" at `docs/planning/orm-state-migration.md:435`. Current PG engine uses `pool_size=5` and `max_overflow=10` at `src/db_pg.py:87-105`. DuckDB docs say read-write mode has "one process" that can read/write and multiple writer threads are within that single writer process at `https://duckdb.org/docs/current/connect/concurrency.html` lines 482-490; the same page says multiple-process write support is through Quack, "in beta stage as of DuckDB v1.5.2" at lines 491-494.
   - impact: A wrong SQLAlchemy pool choice can create extra DuckDB connections, row update conflicts, or multi-process write failures. This is especially risky under FastAPI thread pools and scheduler/app concurrent state access.
   - what the plan should say instead: Specify the DuckDB SQLAlchemy engine factory now: pool class, connection lifecycle on `DATA_DIR` changes, write retry policy, transaction scope, and tests that run app + scheduler-like writes concurrently.

12. Quack timing supports a spike-first strategy, not the current rewrite confidence
   - finding: The plan treats Quack as a medium/variable strategic risk and argues the rewrite can ship before Quack matters.    - evidence: The risk table says "DuckDB-Quack lands fall 2026 and obsoletes the rewrite" at `docs/planning/orm-state-migration.md:437`. The DuckDB concurrency docs say Quack is beta as of DuckDB v1.5.2 and expected mature by v2.0 at `https://duckdb.org/docs/current/connect/concurrency.html` lines 491-494. Local code rejects Quack targets until DuckDB 2.0 at `src/db_state_machine.py:155-161`.
   - impact: Spending 7-9 weeks on a dialect unification that Quack may make unnecessary or reshape is a strategic risk, especially when the lint/repo cleanup value can ship independently.
   - what the plan should say instead: Restructure around invariant enforcement and repo cleanup first, then run the duckdb-engine spike before committing to ORM collapse. Keep Quack as an explicit decision checkpoint with a date and dependency on DuckDB release status.

## VERDICT
- Top 3 plan changes required: First, make Phase 0 a baseline/new-code lint gate instead of a blocking full-repo rule. Second, move the `duckdb-engine` spike before any ORM-collapse commitment and explicitly prove `RETURNING`, Alembic, JSON, pooling, and FK behavior. Third, resolve the usage backend decision so `usage_*` cannot split between DuckDB reads/rollups and PG writes.
- Recommended next step: restructure. Proceed only with repo-method cleanup and baseline linting; spike first for `duckdb-engine`; do not start Phase 8 ORM consolidation until the usage, knowledge-search, chat, and migration-tool gaps above have named contracts and pass/fail gates.
