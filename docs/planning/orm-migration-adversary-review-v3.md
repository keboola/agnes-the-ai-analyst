## v2 finding tracker (11 issues)
1. Phase 7 PG FTS design cannot satisfy its own contract — STILL-BROKEN
   - what v3 does: replaces undefined `unaccent_simple` with `public.cs_unaccent`, drops the BM25-equivalent claim, uses `ts_rank`, and defines lexical parity plus top-10 overlap >= 7 (`docs/planning/orm-state-migration.md:270-305`). The PG grammar pieces check out: `CREATE TEXT SEARCH CONFIGURATION ... (COPY = source_config)` is documented (`https://www.postgresql.org/docs/current/sql-createtsconfig.html` L31-L34, L54-L56); `ALTER MAPPING FOR ... WITH ...` order is documented (`https://www.postgresql.org/docs/current/sql-altertsconfig.html` L29-L32, L57-L59); `unaccent` is a filtering dictionary and the docs show `WITH unaccent, ...` in a mapping (`https://www.postgresql.org/docs/current/unaccent.html` L31, L67-L84); `copy=simple` is shown in current PG docs (`https://www.postgresql.org/docs/current/textsearch-dictionaries.html` L156-L164); `ts_rank` and `ts_rank_cd` are both real (`https://www.postgresql.org/docs/current/textsearch-controls.html` L165-L176, L202).
   - whether it actually resolves the finding: no. The text-search config is valid, but the proposed generated column uses `coalesce(body, '')` (`docs/planning/orm-state-migration.md:287-293`) while the actual table/model column is `content`, not `body` (`src/models/knowledge.py:32-37`, `src/db.py:140-144`, `src/repositories/knowledge_pg.py:307-313`). That DDL is invalid as written. The top-K contract also names a "curated query set" without defining who curates it or what queries are in it (`docs/planning/orm-state-migration.md:272-274`).

3. Phase 6 path b relies on DuckDB-attached PG export not specified enough to gate — CLEAN

(TYPE postgres)`, read-only mode, libpq strings, and PostgreSQL URIs (`https://duckdb.org/docs/current/core_extensions/postgres/overview.html` L481-L491, L497-L512, L559-L562). 4. Shrink-only lint allowlist not implementable as written — STILL-BROKEN

## NEW issues in v3 (if any)
1. Phase 7 DDL uses the wrong knowledge column. v3 proposes `coalesce(body, '')` in the generated `search_vector` (`docs/planning/orm-state-migration.md:287-293`) and says the lexical test matches "body" text (`docs/planning/orm-state-migration.md:272-274`), but the actual table/model/repo use `content` (`src/models/knowledge.py:32-37`, `src/db.py:140-144`, `src/repositories/knowledge_pg.py:307-313`). This makes the Phase 7 migration invalid as written.

2. Phase 2 is still not a complete GO/NO-GO gate. The checklist has only `alembic upgrade head` on a fresh DuckDB (`docs/planning/orm-state-migration.md:155`) and no Alembic autogenerate/idempotence check for duckdb-engine. It also lacks explicit CREATE/ALTER/DROP/type-change DDL coverage, and Alembic is absent from the stop criterion (`docs/planning/orm-state-migration.md:175`).

3. The top-K relevance contract is testable only after methodology is defined. v3 says top-10 results must share >= 7 items for a curated query set (`docs/planning/orm-state-migration.md:272-274`), but names no query set, curator, freeze point, or acceptance methodology. If the implementer writes both the query set and the code, the test is too easy to game.

4. The `state-via-duckdb-attach` security review is underspecified for an LLM SQL surface. The current `/ask` endpoint builds a SQL query from an LLM response and executes it after regex-style SELECT-only validation (`app/api/admin_usage.py:214-288`, `src/usage_ask.py:92-181`). v3 only says "security/performance review" (`docs/planning/orm-state-migration.md:103`) and does not require concrete audit items such as table allowlist, RBAC semantics under attach, function blacklist coverage, timeout/row caps, and logging of rejected/generated SQL.

5. Cost is still under-counted. v3 budgets about 2-4 weeks for 17 cluster-by-cluster ORM PRs (`docs/planning/orm-state-migration.md:432-433`), 0.5 week for Phase 11a's entire migrator regression inventory (`docs/planning/orm-state-migration.md:436`), and 1.5 weeks for roughly 10 Phase 1 quick-win PRs covering about 30 hits (`docs/planning/orm-state-migration.md:423-428`). The next hidden cost is not coding the repo moves; it is making the contract/migrator tests non-shallow and debugging Alembic/duckdb-engine drift.

6. The DuckDB chat "best-effort" public contract lacks a concrete assertion. v3 names `TestDuckDbBestEffortFork` (`docs/planning/orm-state-migration.md:198-201`), and the code says partial failure leaves at most an empty ephemeral session for GC (`app/chat/persistence.py:518-520`), but the plan does not require a failure injection that proves this exact residual state.

## FINAL VERDICT
- Is v3 implementable? yes-but
- Top 3 things still wrong:
  1. Phase 7 is still not implementable as written because it references `body` instead of `content`, and the top-K relevance contract has no curated-set methodology.
  2. Phase 8 still is not a trustworthy registry/FK topo matrix; it uses non-registry names and false FK dependencies.
  3. The gates are still noisy/incomplete: Phase 0 IDs hash the parent function, Phase 2 does not gate on Alembic/DDL autogenerate, and Phase 11 has at least one wrong cited safety range.
- Recommended next step: revise v3 before implementation. Patch Phase 7's column/contract, regenerate Phase 8 from actual `_REGISTRY` keys and model FKs, tighten Phase 0 callsite identity, add Alembic/DDL stop criteria to Phase 2, and correct the Phase 11 cancel/idempotency inventory.
